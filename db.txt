## 1️⃣ USER DOMAIN

| Table / Collection | **users** |
|--------------------|-----------|
| **PK** | `_id : ObjectId` |
| **Columns** | |
| `email` | `String` **required**, **unique**, lower‑cased, validated (regex). |
| `passwordHash` | `String` **required** (bcrypt). |
| `name` | `String` **required**. |
| `role` | `String` enum(`['user','admin','partner']`) default `'user'`. |
| `isVerified` | `Boolean` default `false` (email verification). |

| `phone` | `String` (E.164 format, optional). |
| `address` | Embedded sub‑document (see **address** below). |
| `preferences` | Embedded sub‑document (language, theme, notification opt‑ins). |
| `createdAt` | `Date` default `Date.now`. |
| `updatedAt` | `Date` default `Date.now`. |
| `lastLoginAt` | `Date`. |
| `loyaltyPoints` | `Number` default `0`. |
| `badges` | `[ObjectId]` → **badges** collection (many‑to‑many). |
| `affiliateId` | `ObjectId` → **affiliates** (if user is a creator/partner). |
| `legalConsent` | Embedded sub‑document (`{ termsAccepted: Date, ageVerified: Date }`). |

**Embedded Sub‑document – address**

```json
{
  "street": "String",
  "city": "String",
  "state": "String",
  "postalCode": "String",
  "country": "String"
}
```

**Indexes**

- `{ email: 1 }` unique.
- `{ role: 1 }` (admin lookup).
- `{ "address.country": 1 }` (geo‑filter for legal compliance).

Each file exports a **Mongoose schema** matching the columns above (see the earlier “models” section in the code‑base outline).  

**Example: `User.ts` (excerpt)**

```ts
import { Schema, model, Document } from 'mongoose';

export interface IUser extends Document {
  email: string;
  passwordHash: string;
  name: string;
  role: 'user' | 'admin' | 'partner';
  isVerified: boolean;
  dob?: Date;
  phone?: string;
  address?: {
    street: string;
    city: string;
    state: string;
    postalCode: string;
    country: string;
  };
  preferences?: {
    language: string;
    theme: 'light' | 'dark';
    notifications: boolean;
  };
  loyaltyPoints: number;
  badges: Types.ObjectId[];
  affiliateId?: Types.ObjectId;
  legalConsent?: {
    termsAccepted: Date;
    ageVerified: Date;
  };
  createdAt: Date;
  updatedAt: Date;
  lastLoginAt?: Date;
}

const UserSchema = new Schema<IUser>(
  {
    email: { type: String, required: true, unique: true, lowercase: true },
    passwordHash: { type: String, required: true },
    name: { type: String, required: true },
    role: { type: String, enum: ['user', 'admin', 'partner'], default: 'user' },
    isVerified: { type: Boolean, default: false },
    dob: Date,
    phone: { type: String, match: /^\+?[1-9]\d{1,14}$/ },
    address: {
      street: String,
      city: String,
      state: String,
      postalCode: String,
      country: String,
    },
    preferences: {
      language: { type: String, default: 'en' },
      theme: { type: String, enum: ['light', 'dark'], default: 'light' },
      notifications: { type: Boolean, default: true },
    },
    loyaltyPoints: { type: Number, default: 0 },
    badges: [{ type: Schema.Types.ObjectId, ref: 'Badge' }],
    affiliateId: { type: Schema.Types.ObjectId, ref: 'Affiliate' },
    legalConsent: {
      termsAccepted: Date,
      ageVerified: Date,
    },
  },
  { timestamps: true },
);

UserSchema.index({ email: 1 }, { unique: true });
UserSchema.index({ role: 1 });
UserSchema.index({ 'address.country': 1 });

export const User = model<IUser>('User', UserSchema);
```

Repeat similar patterns for each collection.

## 14️⃣ Data‑Retention & Archival Policy (important for compliance)

| Collection | Retention | Archival Action |
|------------|-----------|-----------------|
| `users` | Keep indefinitely (unless user requests deletion). |
| `orders` | 7 years (legal requirement for financial records). |
| `chat_logs` | 90 days (privacy) → move to **cold‑storage** (e.g., S3 Glacier). |
| `audit_logs` | 1 year (internal QA). |
| `event_logs` | 30 days (monitoring) → aggregate to analytics DB. |
| `design_assets` | Keep as long as the asset is public; if user deletes, purge from S3 after 30 days. |
| `affiliates` | Keep until payout is settled, then archive for 2 years. |
| `delivery_events` | 1 year. |
| `legal_disclaimers` | Keep version history forever (audit). |

Implement a **daily cron job** (Node or Python) that runs the archival logic and removes expired documents.

## 15️⃣ Summary Checklist (what you need to create)

| Step | Action |
|------|--------|
| **1** | Create MongoDB database (e.g., `ganja_goddess`). |
| **2** | Generate all Mongoose models (or SQL DDL) per the tables above. |
| **3** | Add indexes as listed (critical for performance). |
| **4** | Write seed scripts for static data: `categories`, `jurisdictions`, `vendors`, `badges`. |
| **5** | Implement **soft‑delete** (`isDeleted` flag) on collections where legal deletion is not allowed (orders, audit). |
| **6** | Set up **TTL indexes** for `chat_logs` (expire after 90 days). |
| **7** | Integrate **change streams** (MongoDB) or **triggers** (PostgreSQL) to push product/price changes to the **Elasticsearch** sync layer. |
| **8** | Wire the **AI service** to write to `chat_sessions` / `chat_logs` and `audit_logs`. |
| **9** | Add **webhooks** for Stripe, delivery partners, and vendor order confirmations – they will update `orders` status, `delivery_events`, and `affiliate` conversions. |
| **10** | Implement **role‑based access control** (RBAC) at the API layer using the `role` field. |
| **11** | Run **performance load test** (e.g., 10 k concurrent chat sessions) and verify that indexed queries stay < 50 ms. |
| **12** | Document the schema in a **Data Dictionary** (the markdown tables above) and push to your internal wiki. |
